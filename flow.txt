# Step 2 Deep Dive: Parameter Extraction → Code Generation → Security Scanning

## The Real Flow (What Actually Happens)

When a developer says: **"Create a 4 core 8GB RAM instance in staging VPC with only port 443 exposed"**

The system does this:

---

## Phase 0: REQUEST PARSING (BEFORE Security Review)

### Step 0A: Extract Parameters from Request
The system parses the user's natural language request and extracts ALL needed details:

```
User Input: "Create a 4 core 8GB RAM instance in staging VPC with only port 443 exposed"

EXTRACTED PARAMETERS:
├── Instance Type: 4-core, 8GB RAM → maps to EC2 type (e.g., t3.large or m5.large)
├── VPC: "staging VPC" → looks up staging VPC ID from AWS account
├── Network: Staging VPC → determines subnets (public/private)
├── Security Group: Port 443 ONLY → creates security group with INBOUND rule 443/tcp
├── OS: NOT specified → defaults to Amazon Linux 2 (configurable default)
├── Storage: NOT specified → defaults to 20GB gp3 EBS (configurable default)
├── Region: NOT specified → defaults to us-east-1 or account default
├── Tags: NOT specified → applies mandatory tags (Project, Owner, Environment, CostCenter)
└── Other defaults: 
    ├── Monitoring: CloudWatch enabled by default
    ├── IMDSv2: Enforced by default
    ├── Termination protection: Enabled for prod, disabled for dev/staging
    └── Backup/snapshots: Based on environment
```

## Phase 1: GENERATE TERRAFORM CODE

Once all parameters are confirmed, the system generates Terraform code:

### Step 1A: Template-Based Code Generation

The system has **pre-built Terraform templates** for each resource type:

```hcl
# EC2 Template (auto-filled with parameters)
resource "aws_instance" "app_server" {
  # From extracted parameters:
  ami                    = data.aws_ami.amazon_linux_2.id  # OS: Amazon Linux 2
  instance_type          = "t3.large"                       # 4-core 8GB RAM
  subnet_id              = aws_subnet.staging_private[0].id # Staging VPC subnet
  security_groups        = [aws_security_group.app_sg.id]
  
  # Mandatory security hardening:
  metadata_options {
    http_endpoint               = "enabled"
    http_tokens                 = "required"  # IMDSv2 mandatory
    http_put_response_hop_limit = 1
  }
  
  # Root volume (from parameters: 20GB gp3)
  root_block_device {
    volume_type           = "gp3"
    volume_size           = 20
    delete_on_termination = true
    encrypted             = true  # Always encrypt
    kms_key_id            = aws_kms_key.ebs.arn
  }
  
  # Monitoring (from defaults: enabled)
  monitoring = true
  
  # Tags (mandatory from request):
  tags = {
    Name        = "staging-app-server"
    Environment = "staging"
    Project     = "booking-system"  # From user or request
    Owner       = "TeamA"           # From user or request
    CostCenter  = "CC-001"          # From request form
    ManagedBy   = "terraform"
    CreatedBy   = "INFRA-104"       # Jira ticket reference
  }
}

# Security Group (from parameter: only port 443)
resource "aws_security_group" "app_sg" {
  name_prefix = "app-"
  description = "Application security group - port 443 only"
  vpc_id      = data.aws_vpc.staging.id

  # Inbound: Port 443 ONLY (from request parameter)
  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]  # Or restricted based on security review
  }

  # Egress: Allow all outbound (can be restricted based on security review)
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Environment = "staging"
    ManagedBy   = "terraform"
  }
}

# VPC Flow Logs (auto-added by security review)
resource "aws_flow_log" "vpc_flow_logs" {
  iam_role_arn    = aws_iam_role.vpc_flow_logs_role.arn
  log_destination = aws_cloudwatch_log_group.vpc_flow_logs.arn
  traffic_type    = "ALL"
  vpc_id          = data.aws_vpc.staging.id

  tags = {
    Environment = "staging"
    ManagedBy   = "terraform"
  }
}
```

### Step 1B: Security Hardening Applied Automatically

During code generation, the system **applies security best practices**:

```hcl
SECURITY HARDENING CHECKLIST (Auto-Applied):

✓ IMDSv2 Enforcement
  metadata_options.http_tokens = "required"

✓ EBS Encryption
  root_block_device.encrypted = true
  root_block_device.kms_key_id = aws_kms_key.ebs.arn

✓ VPC Flow Logs (if missing from user request)
  → ADDED automatically before security review

✓ CloudWatch Monitoring
  monitoring = true
  cloudwatch_namespace = "CustomApp"

✓ Security Group Least Privilege
  → ONLY port 443 inbound (from user request)
  → All other ports BLOCKED by default

✓ Mandatory Tagging
  → All resources tagged with Project, Owner, Environment, CostCenter

✓ KMS Encryption for Secrets
  → All sensitive data encrypted with customer-managed KMS keys

✓ CloudTrail Logging
  → All API calls logged and auditable
```

## Key Technologies & Tools

### 1. **Request Parsing**
- **Tool**: Natural Language Processing (Claude API, regex patterns)
- **Output**: Structured parameter object

### 2. **Parameter Mapping**
- **Config Database**: JSON/YAML with VPC, subnet, instance type mappings
- **Defaults**: Pre-configured security hardening standards

### 3. **Code Generation**
- **Tool**: Terraform templating (Jinja2, HCL generation)
- **Input**: Extracted parameters + security defaults
- **Output**: Complete, validated Terraform code

### 4. **Security Scanning**
- **Tools**: Checkov, Prowler, TFLint
- **Input**: Generated Terraform code
- **Output**: Compliance report + remediation suggestions

### 5. **Cost Estimation**
- **Tool**: AWS Pricing API or Infracost
- **Input**: Resource types, sizes, regions
- **Output**: Cost breakdown

### 6. **Approval Workflow**
- **Tool**: Jira API + custom backend
- **Input**: Security report, cost breakdown, Terraform code
- **Output**: PR link, approval status

### 7. **Execution & Audit**
- **Tool**: Terraform CLI + CloudTrail
- **Input**: Approved Terraform code
- **Output**: Deployed resources + audit log

## Summary

**HOW THE SYSTEM KNOWS WHERE & WHAT TO DEPLOY:**

1. **WHERE (Region, VPC, Subnets)**
   - Extracted from user request or form
   - Looked up in config database
   - Example: "staging VPC" → vpc-0a1b2c3d4e5f6g7h8, us-east-1, subnets

2. **WHAT (Instance type, OS, storage, specs)**
   - Explicitly stated by user OR
   - Defaults from config file OR
   - Collected via form dialog

3. **SECURITY (Port restrictions, encryption, monitoring)**
   - Extracted from user request (port 443 only)
   - Hardening applied automatically during code generation
   - Validated via security scanning

4. **COST (Calculated before approval)**
   - Based on resource type, size, region
   - AWS Pricing API lookup
   - Shown in approval workflow

All of this happens BEFORE security scanning - the generated code already includes all the details!
